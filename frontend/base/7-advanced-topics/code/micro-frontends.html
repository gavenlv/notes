微前端架构详解

1. 微前端基础概念

微前端是一种架构模式，将前端应用分解为更小、更简单的应用，这些应用可以独立开发、测试和部署，但在客户端被集成在一起，给用户提供统一的体验。

微前端的核心思想是：将大型前端应用拆分为多个小型、松耦合、可独立部署的微应用，每个微应用由不同的团队负责，使用适合其业务的技术栈。

1.1 微前端的优势

  • 技术栈无关：每个团队可以选择适合自己业务的技术栈
  • 独立开发与部署：各团队可以并行开发，独立部署
  • 增量升级：可以逐步升级或替换系统的部分功能
  • 团队自治：小团队可以独立负责完整的业务功能
  • 容错隔离：单个微应用的故障不会影响整个系统

1.2 微前端架构的挑战

  • 应用间通信：微应用之间需要高效可靠的通信机制
  • 共享状态管理：如何在多个微应用间共享和同步状态
  • 路由管理：需要统一的路由系统管理各微应用
  • 样式隔离：避免微应用间的样式冲突
  • 性能开销：可能增加额外的加载和渲染成本
  • 团队协作：需要明确的规范和接口定义

3. 微前端架构实现 - 基于 Qiankun

3.1 Qiankun 框架简介

Qiankun 是一个基于 Single-SPA 的微前端实现框架，由蚂蚁金服开发和维护，提供了更完善的功能和更友好的 API。

主要特性：

  • 基于 Single-SPA 封装，提供了更简单的 API
  • 支持 HTML Entry 接入方式
  • 内置样式隔离方案
  • 提供了 JS 沙箱
  • 支持资源预加载
  • 支持动态加载微应用

4. 微前端核心问题解决方案

4.1 应用间通信

// 基于发布订阅模式的通信方案 // event-bus.js class EventBus { constructor() { this.events = {}; } // 订阅事件 on(event, callback) { if (!this.events[event]) { this.events[event] = []; } this.events[event].push(callback); } // 发布事件 emit(event, data) { if (!this.events[event]) return; this.events[event].forEach(callback => callback(data)); } // 取消订阅 off(event, callback) { if (!this.events[event]) return; if (!callback) { delete this.events[event]; return; } this.events[event] = this.events[event].filter(cb => cb !== callback); } } // 创建单例实例 const eventBus = new EventBus(); // 在主应用中提供给微应用 window.eventBus = eventBus; // 微应用中使用 // 在微应用中订阅事件 eventBus.on('userLogin', (userInfo) => { console.log('用户登录信息:', userInfo); // 更新微应用中的用户状态 }); // 微应用发布事件 eventBus.emit('appInitialized', { appName: 'microApp1' }); // 基于 shared store 的通信方案 // shared-store.js class SharedStore { constructor() { this.state = {}; this.listeners = []; } // 获取状态 getState() { return { ...this.state }; } // 设置状态 setState(newState) { this.state = { ...this.state, ...newState }; this.notifyListeners(); } // 添加监听器 subscribe(listener) { this.listeners.push(listener); // 返回取消订阅函数 return () => { this.listeners = this.listeners.filter(l => l !== listener); }; } // 通知所有监听器 notifyListeners() { this.listeners.forEach(listener => listener(this.getState())); } } // 创建共享状态管理实例 const sharedStore = new SharedStore(); // 在主应用中提供给微应用 window.sharedStore = sharedStore;

│   │   └── app.js             # 应用入口

├── micro-apps/                # 微应用集合
│   ├── micro-app-1/           # 微应用 1
│   │   ├── src/

│   └── theme/                 # 主题样式
└── docs/                      # 文档
    ├── api/                   # API 文档
    ├── guidelines/            # 开发规范
    └── architecture.md        # 架构文档

5.2 团队协作与治理

方面 建议
团队划分 按业务域划分团队，每个团队负责一个或多个微应用
技术栈选择 允许团队选择技术栈，但建议核心框架保持一致
接口规范 制定统一的接口设计规范和 API 文档标准
样式规范 共享设计系统，确保 UI 一致性
代码审查 建立跨团队代码审查机制
监控告警 统一的监控系统，支持按微应用划分
